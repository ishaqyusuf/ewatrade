## Question

What clean relational schema, commands, queries, permissions, idempotency
boundaries, event history, offline envelopes, replay dependencies, media
references, stale-state checks, and conflict responses implement the resolved
Service domain without keeping the current prototype bridge?

## Comments

Design clean Service Request, Quote, Service Job, Job Line, Assignment,
Evidence Reference, Work Event, public token, and Notification Intent
relationships around the shared Order and Service Offering models.

Commands must be tenant-scoped, authorized, idempotent, and guarded by expected
version/state. Offline intake, assignment, evidence capture, and status events
carry stable client IDs and replay dependencies. Conflicting state transitions
return typed conflicts rather than silently overwrite newer work.

Store media references and metadata, not binary files. Enforce public-token
scope and Product-versus-Service separation structurally. Do not retain the
current prototype tables as a compatibility bridge.

## Approved Pipeline Comment

> *This was generated by AI during Wayfinder issue commenting.*

Proposed answer: Introduce clean relations for Request and lines, Quote and
versions, Intake, Job, exact Job Line allocation, Work Event, Assignment
History, Due Commitment, Exception/Rework, Evidence Asset reference, Customer
Tracking Access, Notification Intent, and Delivery Attempt. Provide a dedicated
Service Operations API consuming shared Catalog and Commerce identities. Use
service-specific capabilities. Every mutation is tenant/Store scoped,
idempotent, revision-guarded, and returns typed conflicts. Offline v1 covers
Draft/confirmed Intake, Job/Line status, internal notes, assignment to self,
and evidence capture; public requests, Quotes, payments, publication, and
provider communication require online access. Reset incompatible local Service
state rather than interpreting old events.

Why: First-class relations and versioned command envelopes make invariants,
privacy, replay, and conflicts enforceable without metadata fallbacks.

Assumptions / risk: Media binaries remain outside the relational database;
upload dependencies must complete before publication, not before Intake.
