## Question

What generic repository, service, API, permission, and error contracts should replace business-template and `dryCleaning*` contracts across protected and public flows?

Resolve canonical procedure families, tenant/store/role enforcement, kind validation, compatibility aliases versus clean cutover, public not-found behavior, stable error codes, and the removal boundary for business-template update procedures and metadata authority.

## Comments

> *This was generated by AI during Wayfinder issue commenting.*

Proposed answer: Replace template and `dryCleaning*` contracts with generic families:

- `catalogItems`
- `catalogItemVariants`
- `serviceJobs`
- `serviceRequests`
- `serviceRequestLinks`
- `serviceTracking`
- `serviceOperationsReport`

Remove store-template update and template-mismatch authorization. Authorization remains tenant, store, membership role, and subscription based. Item kind validates whether the requested operation is meaningful.

Use compatibility reads for existing public tokens and deployed clients during migration, but move repository-owned clients to the generic contracts in one coordinated cutover.

Why: Runtime behavior should branch on item kind, never an industry name.

Assumptions / risk: Legacy public URLs may require temporary redirects or token resolvers until existing links expire.
