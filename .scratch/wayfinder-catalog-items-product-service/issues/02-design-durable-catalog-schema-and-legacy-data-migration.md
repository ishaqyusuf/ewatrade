## Question

What durable Prisma model and expand/backfill/contract migration safely moves the existing Product-centered relational catalog and metadata-backed dry-cleaning catalog into the approved item-level model?

## Comments

> *This was generated by AI during Wayfinder issue commenting.*

Proposed answer: Introduce `CatalogItem` and `CatalogItemVariant`, preserving existing physical product tables through database mapping where practical. Add a required item-kind enum and backfill every existing product as `PRODUCT`. Add relational service-job tables and migrate metadata-backed dry-cleaning service items as `SERVICE`.

Use an expand/backfill/contract migration:

1. Add nullable kind and generic service tables.
2. Dual-read existing and new service data.
3. Backfill products and dry-cleaning records idempotently.
4. Verify counts, totals, statuses, tokens, and price snapshots.
5. Switch clients to generic contracts.
6. Make kind required.
7. Remove template and dry-cleaning metadata authority.

Why: This minimizes risky product-table movement while producing a clean domain model.

Assumptions / risk: Legacy service identifiers must be retained in migration mapping for reconciliation.
