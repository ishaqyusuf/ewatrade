## Question

How should inventory, reservations, staff stock wallets, stock intake, unit conversion, sales deductions, returns, closeout, and offline replay enforce that only `PRODUCT` items participate in stock?

Resolve server-side validation placement, database constraints where practical, failure semantics, product creation defaults, and safeguards that prevent a service item from acquiring inventory through direct API calls or legacy replay events.

## Comments

> *This was generated by AI during Wayfinder issue commenting.*

Proposed answer: Every stock operation must load the canonical catalog item and require `kind = PRODUCT` inside the domain/service transaction. Product creation creates inventory records for sellable variants; service creation never does.

Direct API calls, offline replay, stock intake, adjustment, conversion, reservation, staff assignment, return, closeout, and sale deduction must reject services with a stable `ITEM_NOT_STOCKABLE` error.

Why: Hiding stock fields in the interface is insufficient; inventory correctness must be enforced at the write boundary.

Assumptions / risk: Cross-table database constraints cannot express every kind rule, so transactional application enforcement and invariant tests remain necessary.
