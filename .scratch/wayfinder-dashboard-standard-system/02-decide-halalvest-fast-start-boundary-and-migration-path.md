## Question

Which parts of the HalalVest dashboard should EwaTrade copy, port, or avoid, and what migration path keeps the implementation moving quickly while still converging on Midday's architecture?

## Context

The user wants HalalVest used to get started quickly because it already has similar API connection, dashboard, table, sheet, and page patterns. The same request also says EwaTrade must strictly follow Midday for folder/file organization, components, arrangement, pages, performance, tables, analytics, forms, sheets, modals, layout, sidebar, search, and API approach.

This ticket should resolve the tension between fast copying and strict standardization.

## Resolution Target

Decide:

- What is safe to copy from HalalVest as-is.
- What should be ported only after being reshaped to Midday conventions.
- What should be skipped because Midday or current EwaTrade already has a better pattern.
- Whether the first implementation should be a scaffold reset, an incremental route migration, or a hybrid.
- What compatibility shims, if any, are allowed temporarily.
