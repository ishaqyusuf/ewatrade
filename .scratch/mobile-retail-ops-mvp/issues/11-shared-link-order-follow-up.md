The business can complete or cancel the request, record payment method, capture pickup or delivery outcome, and create or update delivery status when needed.

**Status:** implementation-complete

- [x] Customer, admin, and responsible rep receive order-request notifications through the shared email/notification flow.
- [x] Notification failures do not make the customer-facing order submission appear failed after the order has been created.
- [x] Pending order requests appear in the mobile/admin generated-link management surface.
- [x] Follow-up completion can record cash, transfer, or card payment where supported by the existing contract.
- [x] Follow-up completion can capture ready-for-pickup, picked-up, or delivered fulfillment outcome.

## Implementation Notes

- Shared-link checkout already enqueues customer and merchant notifications through `enqueueRetailOpsSharedLinkOrderNotification` and records queued/failed dispatch audit with `recordRetailOpsSharedLinkNotificationDispatch`.
- The storefront order action records notification enqueue failures after the order request is created, so the customer-facing submit flow still redirects with the created request reference when order creation succeeds.
- `apps/mobile/src/components/mobile/product-share-sheet.tsx` shows pending shared-link order requests in the generated-link management surface through `retailOps.sharedLinkOrderRequests`.
- The mobile follow-up controls complete requests with cash, transfer, or card payment and capture ready-for-pickup, picked-up, or delivered outcomes through `retailOps.updateSharedLinkOrderRequestStatus`.
- The Product links sheet now exposes delivery request creation when delivery is selected, collecting compact pickup/dropoff details and calling `retailOps.createDeliveryRequest`.
- The Product links sheet now lists existing delivery requests and supports assigned, picked-up, delivered, and cancelled status updates through `retailOps.updateDeliveryRequestStatus`.
- Existing shared-link DB follow-up logic consumes or releases reservations and updates analytics/rollups when requests are completed or cancelled.
- Added API schema coverage proving generated-link/order/delivery listing filters keep bounded MVP defaults, follow-up accepts supported cash/transfer/card payment and pickup/delivery fulfillment outcomes, delivery creation/status updates accept compact handoff payloads, and unsupported payment, fulfillment, delivery, or malformed delivery details are rejected before business logic.
- Added API schema coverage for optional delivery phone normalization. Delivery request pickup/dropoff phones and assigned-driver phones now trim real input, treat blank phone values as absent, and reject oversized phone values.
- Added focused notification package coverage for shared-link order email dispatch. The tests build the public `createRetailOpsSharedLinkOrderDispatch` payload, plan deliveries, verify one customer confirmation plus owner and sales-rep merchant emails are sent with the expected delivery roles, recipients, order reference, product, total, and follow-up note, and prove a failed sales-rep provider send is captured as one failed delivery while the customer and owner deliveries remain sent. The customer email keeps merchant-facing customer contact details out of the customer copy.
- Added focused DB query coverage for shared-link follow-up completion and cancellation. `packages/db/src/queries/retail-ops-share-links.test.ts` now proves completing a pending request consumes reserved stock, records paid follow-up metadata and receipt details, updates durable request/reservation rows, and rolls up completed/revenue analytics; cancelling a pending request releases reserved stock, records cancelled fulfillment metadata, updates durable request/reservation rows, and rolls up cancellation/released-quantity analytics.
- Added focused DB query coverage for delivery request fulfillment. `packages/db/src/queries/retail-ops-fulfillment.test.ts` now proves creating a durable delivery request stores pickup/dropoff details and mirrors delivery metadata to the order, and marking a durable request delivered writes request status/timestamps, assignment summary, tracking event, and delivered fulfillment metadata.
- Extended focused DB query coverage for delivery follow-up listing. `packages/db/src/queries/retail-ops-fulfillment.test.ts` now proves `listRetailOpsDeliveryRequests` applies actor-scoped durable filters, merges durable delivery rows with metadata fallback requests, de-duplicates fallback rows for the same order, preserves metadata-only delivery rows, sorts by requested time, and returns the compact customer/order/pickup/dropoff/share-link fields used by Product links.
- Added focused API router coverage for authenticated shared-link follow-up. `apps/api/src/trpc/routers/retail-ops-follow-up.test.ts` now proves the protected tRPC router completes a shared-link order request through the tenant/store/user scope, rejects roles without Retail Ops POS access, creates a delivery request for a shared-link order, and updates delivery status with assignment/tracking metadata. The test is included in `qa:mvp-contracts`, and `qa:product-share-management` now protects that contract runner wiring.
- Scoped static checks passed. Android Expo Go QA on `Pixel_3a_API_34` confirmed the dashboard product `Share` action opens the Product links sheet, generated-link management is reachable, and the `Pending link orders` plus `Delivery follow-up` sections render clean production-unavailable fallback states with `Refresh` and `Done` reachable.
