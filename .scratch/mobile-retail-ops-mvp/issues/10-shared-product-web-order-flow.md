# 10 — Shared Product Web Order Flow

**What to build:** a customer opening a shared product link sees a clean web product page with rich preview metadata, chooses a unit or variant and quantity, registers or logs in, and submits an order request. This remains a direct product-link flow rather than a full marketplace or customer app.

**Blocked by:** 09 — Product Share Link Generation And Management.

**Status:** implementation-complete

- [x] Shared links render product preview metadata suitable for WhatsApp and social previews.
- [x] Public product page shows business context, product details, image or fallback visual, available units or variants, and price.
- [x] Customer can choose variant or unit, enter quantity, and see total before submitting.
- [x] First-time customer registration captures name, email, and password.
- [x] Returning customer login supports email and password.
- [x] Order submission creates a pending order request and reserves selected quantity where production inventory supports it.
- [x] Deactivated or invalid links show a clear unavailable state and do not accept new order requests.

## Implementation Notes

- `apps/storefront/src/app/p/[tenantSlug]/[storeSlug]/[productSlug]/page.tsx` resolves public shared products by tenant slug, store slug, product slug, and opaque share token, and returns `notFound()` for missing, invalid, or inactive links.
- `apps/storefront/src/app/p/[tenantSlug]/[storeSlug]/[productSlug]/not-found.tsx` provides the nearest route-level unavailable page, so invalid, missing, expired, or deactivated product links show clear customer-facing copy instead of a generic framework 404.
- The page generates server-side metadata with title, description, canonical URL, Open Graph image, and Twitter card data for rich WhatsApp/social previews.
- First-product setup now captures an optional product description and carries it through production creation, local fallback state, and offline `product_setup` replay, giving the shared product page and rich metadata a real product description source when a new owner creates the first item.
- The public product page shows tenant/store context, product name/description, available units or variants, available quantity, and price.
- `apps/storefront/src/app/p/[tenantSlug]/[storeSlug]/[productSlug]/shared-product-order-form.tsx` now provides the customer order form as a small client component so selected unit/variant and quantity update a visible total before submit.
- First-time customers use the register path with name, email, and password; returning customers use email/password login through the shared auth API before the order request is created.
- Order submission calls `createRetailOpsSharedProductOrderRequest`, creates a pending shared-link order request, records platform customer identity metadata, and reserves selected quantity when a production inventory item supports reservations.
- Deactivated links are now covered on both public lookup paths: durable inactive share links reject before the public product page is shown, and inactive metadata fallback links reject before customer order/reservation writes.
- Hardened the shared checkout form so account mode controls required fields and password autocomplete behavior: first-time registration requires name plus email/password, while returning login uses email/password without making name confusingly required. Quantity now submits a normalized whole-unit hidden value, clamps totals to available stock, disables checkout when no unit is available, and the server action normalizes submitted quantity again before creating the order request. Prompt placeholders use direct instructions rather than sample names, emails, products, addresses, or numeric examples.
- Added API schema coverage proving shared product lookup trims public slug/token inputs, shared order requests normalize customer identity, whole-unit quantity strings are accepted, decimal or zero quantities are rejected, and short public tokens are rejected before the order request reaches business logic.
- Extended API schema coverage for shared-link customer email normalization. Shared product order requests now trim and lowercase customer emails before validation while still rejecting invalid email formats.
- Extended API and storefront coverage for optional customer phone handling. Shared product order requests now trim phone input, treat blank phone values as absent, reject oversized phone values, and the storefront server action trims phone and notes before creating the pending order request.
