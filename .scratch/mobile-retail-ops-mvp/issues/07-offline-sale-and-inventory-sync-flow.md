# 07 — Offline Sale And Inventory Sync Flow

**What to build:** after a successful login, users can keep working when offline, queue supported sales and inventory events safely, and sync them idempotently when connectivity returns. The user should always understand when changes are local, syncing, synced, failed, or conflicted.

**Blocked by:** 05 — Create Sale With Product Variants And Quantity Stepper; 06 — Customer Book During Checkout.

**Status:** implementation-complete

- [x] Offline mode is available only after a successful authenticated session and business context have been established.
- [x] A visible offline banner appears while offline and does not block create-sale.
- [x] Sales, customer upserts, stock intake, unit conversion, manual adjustment, session, closeout, staff invite, and share-link events use durable local queue records where supported by the MVP.
- [x] Queued events carry client event id, tenant, store, actor, device, event type, payload, dependencies, retry state, and timestamps.
- [x] Reconnect sync replays events without creating duplicate sales, customers, or inventory movements.
- [x] Applied sync results reconcile production ids into local sale, customer, product, session, staff, and share-link records.

## Implementation Notes

- `apps/mobile/src/store/retailOpsStore.ts` now stores richer local sync envelopes for queued MVP events, including the client event id, business/tenant id, optional store id, actor name when known, offline device id, event type, entity id, dependency metadata, retry count/state, created timestamp, and updated timestamp.
- Queue dependencies now cover product/unit prerequisites, cashier sessions, sales, customers, staff invites, share links, and stock/unit-conversion records where those entities affect replay order or production id resolution.
- The existing persisted local records remain the source of truth for replay payloads. `apps/mobile/src/lib/retail-ops-sync.ts` derives typed tRPC payloads at sync time so product, unit, session, customer, share-link, and staff remote ids can be filled after earlier dependency events apply.
- Existing offline replay uses event ids as external idempotency keys for `product_setup`, `sale_created`, `customer_upsert`, session, closeout, staff, share-link, stock-intake, stock-adjustment, and unit-conversion events.
- Existing sync resolution maps applied production ids back into local product/customer/session/sale/staff/share-link rows, marks stock movements as synced through movement ids or conversion group ids, records failed/conflict state, supports retry backoff, and clears failure state after successful retry.
- The dashboard and sync sheet already expose offline mode, pending/synced/failed/conflict state, supported event counts, retry actions, server sync history, and manager-visible sync conflicts without blocking create-sale while offline.
- `apps/mobile/scripts/check-session-offline-persistence.mjs` now guards the core after-login offline contract: app sessions stay in `expo-secure-store`, mobile tRPC calls forward the stored app bearer token through `x-app-authorization`, the API context accepts that bearer header, and Retail Ops business state plus sync events persist through the shared Zustand storage adapter.
